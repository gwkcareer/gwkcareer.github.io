# 프로세스 vs 스레드


"스레드 풀", "멀티프로세스", "컨텍스트 스위칭" 같은 단어를 매일 마주친다. 
그런데 정작 프로세스와 스레드의 차이를 한 줄로 설명하라고 하면 막히는 경우가 많다. 핵심만 정리해본다.


## 한 줄 정의

- **프로세스(Process)**: 실행 중인 프로그램. OS로부터 독립된 메모리 공간을 할당받는 실행 단위.
- **스레드(Thread)**: 프로세스 안에서 실제로 일을 하는 실행 흐름. 한 프로세스 안의 스레드들은 메모리를 공유한다.

비유하자면 프로세스는 "사무실(독립 공간)", 스레드는 "그 사무실에서 일하는 직원들"이다. 
직원들은 같은 사무실의 책상·자료(메모리)를 공유하지만, 각자 자기 손에 든 작업 노트(스택)는 따로 가진다.


## 메모리 구조가 핵심이다


둘의 차이는 결국 **메모리를 어떻게 나눠 갖느냐**로 귀결된다.


| 메모리 영역                    | 프로세스      | 스레드                    |
| ------------------------- | --------- | ---------------------- |
| Code / Data / **Heap**    | 프로세스마다 독립 | 같은 프로세스 내 스레드끼리 **공유** |
| **Stack**                 | 프로세스마다 독립 | 스레드마다 **독립**           |
| PC(Program Counter), 레지스터 | 독립        | 스레드마다 **독립**           |


스레드가 Heap을 공유한다는 점이 멀티스레딩의 모든 장점(빠른 통신)과 단점(동기화 문제)의 출발점이다.


## 비교 정리


| 구분       | 프로세스                       | 스레드                         |
| -------- | -------------------------- | --------------------------- |
| 메모리      | 독립 공간                      | Heap 등 공유, Stack만 독립        |
| 통신 방식    | IPC 필요 (파이프, 소켓, 공유메모리 등)  | 공유 메모리로 직접 통신               |
| 생성 비용    | 큼 (메모리 전체 할당/복사)           | 작음                          |
| 컨텍스트 스위칭 | 무거움 (메모리 맵 교체 등)           | 가벼움                         |
| 안정성      | 한 프로세스가 죽어도 다른 프로세스는 영향 없음 | 한 스레드의 오류가 프로세스 전체를 죽일 수 있음 |


프로세스는 **격리(안정성)**, 스레드는 **효율(속도·비용)**을 가져간다.


## 멀티스레딩의 양날의 검: 동기화


스레드들이 Heap을 공유하기 때문에, 여러 스레드가 같은 데이터를 동시에 건드리면 **race condition**이 발생한다. 
그래서 `synchronized`, `Lock`, `Atomic`, 그리고 메모리 가시성을 위한 `volatile` 같은 동기화 도구가 필요하다.


반대로 프로세스는 메모리가 분리돼 있어 이런 문제에서 자유롭지만, 데이터를 주고받으려면 IPC라는 별도 비용을 치러야 한다.


## 그럼 스레드 문제는 어떻게 해결하나


해결 전략은 결국 둘 중 하나다. **문제를 아예 피하거나(회피)**, **잘 관리하거나(제어)**


### 1단계. 가장 좋은 해결책은 "공유하지 않는 것" (Stateless)


스레드 문제의 근본 원인은 공유 자원, 즉 **Heap에 있는 객체 필드**다. 애초에 공유할 상태를 만들지 않으면 동기화를 고민할 일이 없다.

- **지역 변수 활용**: 메서드 안에서 선언한 변수는 스레드별 Stack에 저장되므로 다른 스레드가 침범할 수 없다. 가장 안전한 형태다.
- **불변 객체(Immutable Object)**: 한 번 만들면 수정 불가능하게 설계한다. (`final` 필드, Setter 제거) 읽기만 가능하면 여러 스레드가 동시에 접근해도 안전하다.

### 2단계. 자바의 "검증된 도구" 활용 (Concurrent Library)


`synchronized`나 `Lock`을 직접 다루는 건 실수 확률이 높다. 표준 라이브러리가 제공하는 고성능 도구를 먼저 쓰는 게 정석이다.

- **Atomic 클래스**: `AtomicInteger`, `AtomicLong` 등. CAS(Compare-And-Swap) 기반 CPU 명령 수준 최적화라, `synchronized` 없이도 숫자 연산을 빠르고 안전하게 처리한다.
- **Concurrent Collections**: `ArrayList` 대신 `CopyOnWriteArrayList`, `HashMap` 대신 `ConcurrentHashMap`. 내부적으로 스레드 안전성을 보장한다.
- **ThreadLocal**: "공유는 하되 스레드마다 다른 값을 유지"하고 싶을 때. (예: 요청별 로그인/트랜잭션 컨텍스트) Spring의 `SecurityContextHolder`, 트랜잭션 동기화가 이 방식이다.

### 3단계. 피할 수 없는 동기화는 "최소화" (Locking Strategy)


정말 여러 스레드가 하나의 자원을 수정해야 한다면

- **범위 최소화**: 메서드 전체에 `synchronized`를 거는 대신, 꼭 필요한 몇 줄에만 Lock을 건다. Lock 구간이 짧을수록 경합이 줄어 처리량이 올라간다.
- **데드락 방지(Lock 순서 고정)**: 여러 Lock을 잡아야 한다면 **모든 스레드가 항상 같은 순서로** 획득하게 설계한다. 
(A → B로 정했으면 어디서든 A를 먼저 잠근다.)

### 정리: 우선순위 마인드셋

1. **가장 먼저** → 공유 자원 자체를 없앤다 (Stateless).
2. **안 되면** → `Atomic` / `Concurrent` 라이브러리를 쓴다.
3. **그래도 안 되면** → `synchronized` / `ReentrantLock`을 쓰되 범위를 최대한 좁게 잡는다.

이 셋을 상황에 맞게 조합하면 스레드 문제의 대부분은 해결된다.


## 한 줄 요약

> 프로세스는 격리를 위해 메모리를 나누고, 스레드는 효율을 위해 메모리를 공유한다. 공유하는 만큼 빠르지만, 공유하는 만큼 동기화가 필요하다.   
> 그래서 해결의 1순위는 동기화를 잘하는 게 아니라 **애초에 공유하지 않는 것**이다.
